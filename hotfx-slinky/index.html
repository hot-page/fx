This all started when I found an awesome CodePen by TK. As happens to me all the time these days, I thought to myself, "That could be a web component!" And a few lines of code later, the <hotfx-slinky> element was born. Now any text can be bouncing rainbow text.

I was pretty pleased with myself at this point, but then I thought, people are going to want even more. And by people I mean myself. For instance, I already wanted horizontal movement instead of vertical movement. I was about to add another 5 variables and some more animations, but then I thought, this is never going to be enough for me....

Instead of adding more variables, I should just let people write their own CSS. Now we can get horizontal movement, or movement in three dimensions or whatever else you want.

But wait, let's back up for a second: What is really going on here? At it's most basic level, this is a custom element that duplicates it's children a bunch of times and then gives colors to them as a CSS variable.

What if we just throw some attributes on this puppy to control all of that? Then we can really open up the possibilities.

In the demo below, you can control the number of elements created with the n attribute and change the start and end colors for the gradient. The interpolate-long attribute takes the long way around the color wheel, so if you want a rainbow, that's your ticket.

So far we've been playing with the same text. Obviously you can change the text, but you can also use any DOM element. Here we've got a circle of burning poop. Delightful.

Shout out to Yuan Chuan and his <css-doodle> element for paving the way here. CSS doodle makes a give number of empty DOM elements which you can then change however you want with a crazy custom CSS syntax.

Is this even a good idea?

Obviously, the answer is yes, because it has been a lot of fun. But there's a more interesting question: should this even be a custom element? Well, maybe not. Instead of putting one h1 tag inside a <hotfx-slinky> and setting n=20, we could achieve the exact same thing by just copy/pasting 20 <h1> tags in HTML. Then our page and wild slinkies would work without any JavaScript at all.

But let's face it, who has time to be a purest? Writing this as a custom elements makes things just be a bit easier. For one, the element makes our HTML much easier to read and modify. If you want to go from 10 copies to 90 you're just changing one number instead of managing a tonnnn of markup.

Also these attributes can be changed on the client and the element will re-render it's DOM. Really that's the only way to make this thing dynamic like we did with the attribute controls above.

Too much JavaScript or reliance on it when you don't need it is bad. But in this case, we are only relying on a little bit of JavaScript (about 100 lines or so). Your users will surely never notice.
